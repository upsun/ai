## Gatsby Application Configuration

Reference configuration for Gatsby static site applications on Upsun.

**Framework Overview**: React-based static site generator with SSR support and dynamic data fetching capabilities.

YAML example (minimal, production-ready):

```yaml
applications:
  site:
    type: nodejs:22

    build:
      flavor: none

    hooks:
      build: |
        set -ex
        npm install --omit=dev
        npm run build

    web:
      commands:
        start: npm run serve -- --port=$PORT

      locations:
        /:
          root: public
          passthru: true
          index: [index.html]

    mounts:
      .cache/data:
        source: tmp
```

Configuration details and usage notes:

- Node.js Version: use nodejs:22 as the runtime in Upsun.
- Source root: if your Gatsby site lives in a subdirectory, add a `source` block under the application. Example:

```yaml
applications:
  site:
    source:
      root: path/to/site
    type: nodejs:22
    # ...rest as above
```

- Package installation: the example uses `npm install --omit=dev`. For deterministic production installs when you have a reliable package-lock.json, prefer `npm ci --omit=dev`.
- Build process: runs the Gatsby build script to produce the static site into the `public/` directory.
- Runtime server: serves pre-built files using `gatsby serve` (invoked here via an npm script `serve`). Ensure your package.json exposes the intended `serve` script.
- Static files: Upsun serves from the `public/` directory generated by the build.
- Cache management:
  - Mount `.cache/data` to `tmp` to provide writable runtime cache storage.
  - Avoid mounting the entire `.cache` directory; mounting only `.cache/data` preserves build-time cache files and avoids unintended side effects.
- Port configuration: the runtime uses the Upsun-provided `PORT` environment variable.

Gotchas:
- Only use `npm ci` when package.json and package-lock.json are in sync; otherwise fall back to `npm install` to avoid install failures.
- Confirm your `npm run serve` script accepts a `--port` option or adapt the start command to your server implementation.
